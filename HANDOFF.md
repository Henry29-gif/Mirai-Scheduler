## Core Features
### Employee (STAFF)
- View own schedule; accept open shifts; **Trade** (rest-safe) and **Drop** to open board (no "Sick" for staff).
- Time-off requests; weekly availability grid; time clock + personal timecard; in-app notifications.
- My Space: Request time off · My availability · My timecard · **Certification** (records + expiry; own cert documents — web in progress, mobile pending).
- Forgot/reset password; delete account (erases PII).

### Admin / Manager
- Auto-scheduler (RN/LPN/CCA) honoring **staff availability** + approved time-off (verified: a staffer marked unavailable for a weekday gets zero shifts that day), manager-set **per-shift, per-calendar-date** staffing counts (a box for every date in the month — `StaffingRequirement.date` `@db.Date`, **changed 2026-06-23 from per-weekday**; old per-weekday rows were cleared in the migration, and the express json body limit was raised 10kb→100kb for the bigger grid payload; "Copy to all days" fills the month; verified: Jun 15 Day-RN=4 → 274 total shifts vs 270 baseline), 8h rest (doubles allowed, no triples, no weekly cap), fair distribution; **date-range** generation. (Removing the weekly cap let understaffed sites like Lakeside fill all 7 days incl. weekends.)
- **Draft-schedule calendar** (month grid + week-view toggle): each day shows RN/LPN/CCA across Day/Evening/Night as color-coded, draggable chips. **Drag a name onto a coworker → swap**; **drag onto an Open slot → fill it** (their old slot reopens). Same-role only; 8h-rest enforced; cross-role/rest-breaking drops rejected with a friendly message. Backed by `POST /api/schedules/move {sourceShiftId,targetShiftId}`. (Replaced the old flat table for managers.)
- Review-before-post: **Distribution preview** (per-person load + fairness spread) → drag-and-drop balance → **Post** (publish). The reassign-candidates endpoint still exists as a fallback.
- Open-shift board (ranked candidates, assign, release); **Live attendance**; **Timecards** (approve/reopen + missed-punch fix); **Time-off approvals**.
- **Shift-activity alerts (2026-06-23)**: whenever a staff member picks up, drops, or calls in sick on a shift, requests/accepts a trade, or requests time off, the staff member's supervisors get an in-app notification (bell on web + mobile, polled every 30s). Recipients = MANAGERs at that staff member's facility + ADMINs across their organization (the actor is excluded). Backend-only change: helpers `supervisorIdsForStaff`/`notifySupervisors` in `services/notify.service.ts`, called from `shift.routes.ts` (accept, release), `swap.routes.ts` (request, complete), `timeoff.routes.ts` (request). Deliberately NOT wired to availability-grid toggles or clock-in/out (too noisy; clock activity has the Live-attendance view).
- **Notification quick actions (2026-06-23)**: request-type notifications carry `metadata {kind,id}` (`TIMEOFF_REQUEST` set via `notifySupervisors`; `SWAP_REQUEST` set via `notify` to the target staff) and render inline **Accept / Decline** buttons in the bell (web + mobile). Accept→approve/accept, Decline→deny/decline — one tap, no confirm — hitting `/api/timeoff/:id/respond` or `/api/swaps/:id/respond`. `notify`/`notifyMany`/`notifySupervisors` gained an optional `metadata` arg; both front-ends track acted notif ids and swap the buttons for a result label ("Accepted ✓" / "Declined" / "Already handled"). ## Business Rules
- Skill match (shift requires a certification); **8-hour rest between work blocks; doubles (two back-to-back shifts = 16h) allowed, triples (24h) blocked; NO weekly-hours cap** (centralized in `backend/src/utils/rest.ts` → `checkRest`, used by the generator, calendar drag-drop, reassign, and staff pickup). Overtime *pay* (1.5× over 40h/wk) is still tracked for cost projections only.
- Scheduler fills manager-defined per-shift counts (default 1 each); unfilled → OPEN; fair distribution (fewest accumulated hours, tie-break = worked-longest-ago).
- **Draft schedules are admin-only**: STAFF see neither assigned nor OPEN shifts until the period is **PUBLISHED**. Generating replaces the whole period (must re-post).
- **Pay/hourly rate = ADMIN-only**, enforced server-side.
- **Multi-tenant isolation**: all facility-scoped queries go through `utils/tenant.ts`; an ADMIN sees only their Organization; cross-tenant access → 403.
- **Certifications**: staff manage their own, admins view read-only. **Cert documents**: visible only to the owner + their managers/admins, never to other staff; staff cannot see admin-uploaded HR docs (`StaffDocument.source = STAFF` vs `ADMIN`).
- Account deletion anonymizes PII but keeps de-identified work records. Billing is B2B / out-of-app (no in-app purchases).
- **Auto sign-out after 7 minutes of inactivity** (web + mobile; `IDLE_MS` in each app root). Any interaction resets it. Web persists `ns_last_activity` so a tab reopened after the window lands on login; mobile re-checks on returning to the foreground. (Separate from the still-pending `expo-secure-store` token-at-rest work.)

## Security posture (audited 2026-07-03 vs a 6-point checklist; fixes applied + adversarially re-verified)
- **Passwords**: bcrypt cost 12 everywhere (set + verify), no plaintext anywhere, reset tokens random/sha256-hashed in Redis/1h-TTL/single-use/no-enumeration. PASS.
- **Session revocation (NEW)**: `authenticate` now checks (1) a Redis `pwchanged:<userId>` stamp — any JWT issued before a password reset or account deletion is rejected (verified live: old token 401s the instant the password resets) — and (2) `isActive` in the DB (deactivated accounts die on next request). Socket handshake mirrors both. Stamps are written by reset-password + account-delete. DB errors → 500 (not 401) so a flaky DB can't force logouts. Known accepted edge: a token minted in the same 1-second window as the stamp escapes (needs attacker login in that exact second).
- **Redis on the request path uses a dedicated fast-failing client** `redisAuth` (`enableOfflineQueue:false`, 1s command timeout) in `config/redis.ts` — the BullMQ client (`maxRetriesPerRequest:null`) queues forever during outages and must NEVER be used in middleware (verified live: API answers in ~34ms with Redis stopped; would previously hang).
- **Clients auto-logout on 401** (web reload → login; mobile `onSessionExpired`) — only when the rejected token is still the current one (stale in-flight 401s can't kill a fresh session). Login itself unaffected (sends no token). Accepted minor: a few raw-fetch paths (CSV export, doc up/downloads) bypass this and self-heal via the 30s notification poll.
- **Transport/headers**: helmet defaults (CSP, HSTS 180d) ✓; `app.set("trust proxy", 1)` added (without it, express-rate-limit v7 behind Render keys everyone to one bucket → global login lockout); Express + Socket.io CORS now share `utils/origins.ts` (trimmed allowlist, dev-localhost fallback, never `*`); cookie-parser removed (no cookies exist — pure Bearer JWT).
- **Email**: in production without RESEND_API_KEY, `sendEmail` now throws instead of logging the message body (reset links contain live tokens — never in logs). Dev log-mode unchanged.
- **Boot guard**: production refuses to start if JWT_SECRET is missing/<32 chars/placeholder. **Sentry**: request bodies stripped from error events (`requestDataIntegration({include:{data:false}})`).
- **At rest / keys**: `.env` gitignored + never committed (verified incl. history); no hardcoded secrets repo-wide; render.yaml uses generateValue/sync:false. Field-level DB encryption + E2EE assessed and deliberately skipped (workforce data, not patient data; platform disk encryption on Render + RBAC is the bar). Deleted dead files: `controllers/auth.controller.ts` (contained an unauthenticated register handler), `clock.controller.ts`, `cron.service.ts`.
- **Launch reminders from the audit**: remove demo-credential prefill from both login screens before prod builds; never run `db:seed` against prod; set ALLOWED_ORIGINS/APP_WEB_URL/RESEND_API_KEY in Render dashboard (blueprint comment updated: Resend key is REQUIRED in prod).
